# DATABASE_SCHEMA.md

This file describes the database model that the frontend must understand. The frontend does not directly access the database, but it must respect the available entities, fields, relations, and workflows.

Do not invent fields that are not listed here unless the backend already returns them.

---

# 1. organisations
Represents legal organizations that apply to become contributors.

## Fields

```ts
type Organisation = {
  id: string; // UUID
  name: string;
  siret: string;
  email: string;
  website?: string | null;
  description?: string | null;
  country?: string | null;
  api_key_hash?: string | null; // backend only, never displayed as plain API key
  trust_score: number; // 0-100
  status: 'pending' | 'approved' | 'revoked';
  created_at: string;
};
```

## Notes for frontend

- The Become Contributor form creates an organization request with status `pending`.
- Admin can approve/revoke organizations depending on backend endpoints.
- On approval, backend generates an API key and creates a contributor user.
- The API key is stored hashed. Do not expect to read the plain API key from the database.
- The plain API key should only be available when generated/regenerated by backend.

## Relations

- One organization can submit many IOCs.
- One organization can submit many threat actors.
- One organization can submit many malware samples.
- One approved organization has one contributor user account.

# 2. contributor_users
Represents web portal accounts for approved contributors.

This is now part of the official frontend logic, not optional.

## Fields

```ts
type ContributorUser = {
  id: string; // UUID
  org_id: string; // FK -> organisations.id, unique
  email: string;
  hashed_password?: string; // backend only, never displayed
  must_change_password: boolean;
  is_active: boolean;
  created_at: string;
};
```

## Notes for frontend

- A contributor user is created automatically when admin approves an organization.
- The contributor receives login email/username and temporary password by email.
- On first login, `must_change_password` may be true.
- If `must_change_password = true`, redirect to `/contributor/change-password`.
- Contributor users log in from the same `/login` page as admin.
- Contributor user password is for the web dashboard only.
- Contributor API key is for automation only.

# 4. iocs
Represents indicators of compromise.

## Fields

```ts
type IOC = {
  id: string; // UUID
  type: 'ip' | 'url' | 'hash' | 'email';
  value: string;
  description: string;
  org_id: string;
  tlp: 'red' | 'amber' | 'green' | 'white';
  confidence: number; // 0-100
  first_seen?: string | null;
  last_seen?: string | null;
  tags: string[];
  source_context?: string | null;
  status: 'validated' | 'pending' | 'rejected' | 'revoked' | 'false_positive' | 'deprecated';
  submitted_at: string;

  // enrichment fields, if backend returns them
  country_code?: string | null;
  asn?: string | null;
  abuse_score?: number | null;

  // optional relations returned by some endpoints
  organisation?: Organisation;
  blockchain_records?: BlockchainRecord[];
};
```

## Frontend rules

Public pages:

- Show only `tlp = green` or `tlp = white`.
- Show only public/validated data returned by backend.

Contributor dashboard:

- Show only IOCs submitted by contributor's organization.
- Contributor can create new IOCs.
- Contributor cannot edit IOCs.
- Contributor cannot delete IOCs.
- Contributor can mark their own existing IOC as false positive.

Admin dashboard:

- Can view all IOCs if backend supports it.
- Can inspect pending/suspicious IOCs if backend supports it.

# 6.1 Generic false-positive status rule

The frontend must treat false positive as an immutable correction action, not as edit/delete.

Applies to contributor-owned records:

- IOCs
- Malware samples
- Threat actors

Rules:

- Show a `Mark as false positive` action only for records owned by the authenticated contributor's organization.
- Do not show this action for records already in `false_positive`, `rejected`, or `revoked` status.
- Backend must verify ownership before accepting the action.
- If backend does not yet support false positive for malware/threat actors, keep the UI action hidden behind a capability flag or TODO, but keep the schema prepared for `false_positive`.

# 10. Frontend security rules

- Never store hashed passwords in frontend state.
- Never display `api_key_hash`.
- Never hardcode admin credentials.
- Never store API keys in frontend unless there is a clear backend feature for that.
- Store JWT carefully; if existing project already uses localStorage, follow project convention. Prefer an auth abstraction so storage can be changed later.
- All protected API calls must include `Authorization: Bearer <token>`.
- Automation API calls, if implemented manually, use `X-API-Key`, but normal contributor dashboard should prefer JWT endpoints.
